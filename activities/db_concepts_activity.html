Concept Overview Understanding DBMS Fundamentals

Data is raw, unorganized facts (e.g., "Ram", "9841-234567"). Information is data that has been processed and organized to make it meaningful (e.g., a student directory). A database is a structured collection of data stored electronically. A DBMS is software that manages the database.

Real-world example: An electricity authority uses a database to manage customer billing, outages, and payments. Similarly, schools use databases for student records.

Key Concepts in DBMS:

  • PK Primary key -- uniquely identifies each record (e.g., Student_ID).
  • FK Foreign key -- links to primary key of another table (e.g., Dept_ID in Student table).
  • ER Entity-Relationship (ER) model -- a diagram that shows entities (tables) and their relationships.

Common Mistakes to Avoid:

  • Confusing data with information -- data is raw, information is processed.
  • Thinking a database is the same as a DBMS (DBMS is the software that manages the database).
  • Forgetting that foreign keys must reference a primary key in another table.

Best Practices:

  • Always identify the primary key first when designing a table.
  • Use meaningful names for tables and columns.
  • Draw ER diagrams before implementing the database schema.

Part 2 Keys in Relational Databases

Activity 2.1 -- Identify the Key Type

Given the Student table with columns: Student_ID, Roll_No, Name, Dept_ID, Phone

  1. Primary key -- Student_ID (unique, not null)
  2. Candidate key -- Roll_No (also unique)
  3. Foreign key -- Dept_ID (references Department table)
  4. Composite key -- (Student_ID, Phone) if used together to identify a record
  5. Super key -- any set that contains a candidate key, e.g., (Student_ID, Name)

Activity 2.2 -- Alternate Key

In the Employee table, Emp_ID is the primary key. What is the alternate key if PAN_Number is also unique?

PAN_Number is an alternate key.
Part 3 ER Diagrams & Relationships

Activity 3.1 -- Relationship Types

Identify the relationship type for each scenario:

  1. A college has many students, but a student studies in only one college. Many-to-One (M:1)
  2. A person has one passport, and a passport belongs to one person. One-to-One (1:1)
  3. A student can enroll in many courses, and a course can have many students. Many-to-Many (M:N)

Activity 3.2 -- ER Diagram Symbols

Match each shape to its meaning in an ER diagram:

  • Rectangle = Entity
  • Ellipse = Attribute
  • Diamond = Relationship
  • Double ellipse = Multivalued attribute (e.g., phone numbers)
  • Dashed ellipse = Derived attribute (e.g., age from DOB)

Example: In a Student entity, Student_ID is a key attribute (underlined), Address is composite (Province, District, Municipality), and Phone is multivalued.

Activity 4.2 -- Database Security Threats

List three common security threats:

  • Insider threats -- employees with too much access
  • SQL injection -- malicious code inserted into queries
  • Denial of Service (DoS) -- overwhelming the server with requests

Best practice: Use encryption, regular patching, and least privilege access.
